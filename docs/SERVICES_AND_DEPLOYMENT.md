# Hilal Vision — Services & Deployment Guide

This document details the external services, infrastructure, APIs, and deployment pipelines powering the Hilal Vision ecosystem (Web + Mobile).

---

## 1. Deployments & Infrastructure

### Web Application (Next.js 15)
- **Host:** Vercel
- **Branch Strategy:** Commits to `main` trigger automatic production builds.
- **Caching (`@vercel/kv`):** Used strictly for caching heavy astronomical calculations (like the global Yallop visibility heatmaps). Next.js App Router caching (`fetch` caching) handles localized static data.
- **Edge Deployment:** tRPC API routes are deployed to Vercel Edge functions for low-latency responses, particularly the telemetry data intake.
- **Required Env Vars on Vercel:** See the `.env.example` file. Includes Clerk keys, Stripe keys, and Neon database URLs.

### Mobile Application (Expo / React Native)
- **Framework:** Expo SDK + Expo Router
- **Build System:** EAS (Expo Application Services)
- **Deployment Strategy:**
  - **Android (`.aab`):** Built via `eas build -p android --profile production` and submitted to Google Play Console.
  - **iOS (`.ipa`):** Built via `eas build -p ios --profile production` and submitted to Apple App Store Connect via Transporter.
  - **OTA Updates:** Managed via Expo Updates for pushing JS/asset patches without full app store reviews.
- **Architecture Note:** The mobile app heavily reuses the `@hilal/astronomy` package to perform on-device calculations seamlessly without network dependencies.

## 2. External APIs & Services

### Authentication & User Accounts (Clerk)
- **Service:** `@clerk/nextjs` (Web) and `@clerk/clerk-expo` (Mobile)
- **Implementation:**
  - Route protection is managed centrally in `middleware.ts`.
  - User identity is handled entirely by Clerk. We do *not* store passwords or manage sessions locally.
  - Custom user metadata (`user.publicMetadata`) stores flags for user permissions (`isPro`, `isPatron`, `isAdmin`).
  - The `ProTierSync` component synchronizes Clerk metadata into the Zustand store for seamless UI updates.

### Payments & Subscriptions (Stripe)
- **Service:** Stripe Checkout
- **Flow:**
  1. User triggers `UpgradeModal`.
  2. Next.js API route (`/api/stripe/checkout`) generates a Stripe Checkout Session tied to the user's Clerk ID.
  3. Upon successful payment, Stripe fires a webhook to `/api/stripe/webhook`.
  4. The webhook verifies the signature and updates the user's `isPro` status directly in the Clerk dashboard using the Clerk Backend API.
- **Tiers:** Monthly ($5), Annual ($50), Lifetime ($150).

### Database (Neon PostgreSQL + Drizzle ORM)
- **Service:** Neon (Serverless PostgreSQL)
- **Key Feature:** We rely heavily on **PostGIS** for storing and querying geographical observation data.
- **Connection:** Uses the Neon HTTP driver (`@neondatabase/serverless`) which eliminates connection pooling limits during Vercel Edge cold starts.
- **Location Storage:** Observation reports are stored with a geometry `Point` column for efficient spatial querying (e.g., finding sightings near a specific latitude/longitude).

### Weather Data (Open-Meteo)
- **Service:** Open-Meteo API
- **Usage:** While the primary `/weather` route was archived, weather data is integrated into the visibility algorithms to calculate the "Best Time to Observe" by factoring in localized cloud cover predictions during Sunset/Moonset windows. No API key is required; it's a free, open-source weather API.

### Maps (MapLibre GL JS)
- **Service:** MapLibre
- **Usage:** Rendering the interactive crescent visibility heatmap. The frontend generates GeoJSON polygons based on the Yallop criteria array generated by the astronomy engine and overlays them onto a MapLibre base layer.

## 3. SEO & Discoverability

### Next.js App Router Metadata
- **Static Metadata:** Defined in the root `layout.tsx` (`title`, `description`, `applicationName`, etc.).
- **Dynamic Metadata:** Pages like `/moon` and `/calendar` dynamically generate metadata based on the current Hijri month or astronomical event to target highly specific long-tail keywords (e.g., "Shawwal 1445 moon sighting visibility").
- **Sitemap & Robots.txt:** Placed in the `app` directory to generate standard SEO crawl indexing files automatically.
- **i18n URLs:** English, Arabic (`/ar`), and Urdu (`/ur`) locales ensure content is properly indexed across global regions using built-in Next.js i18n features.

## 4. Testing Strategy

### E2E Testing (Playwright)
- **Scope:** Playwright handles critical user flows including:
  - Auth flows (mocking Clerk).
  - Stripe checkout redirections.
  - Ensuring the 3D globe and visibility map render successfully.
  - Validating "I saw it!" telemetry form submissions.

### Unit Testing (Vitest)
- **Scope:** Tests focus almost entirely on the `@hilal/astronomy` package.
- **Goal:** Ensure mathematical accuracy against established data (e.g., verifying Yallop q-values and conjunction times exactly match HMNAO published data).
